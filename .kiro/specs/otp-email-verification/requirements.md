# Requirements Document

## Introduction

Add OTP (One-Time Password) email verification to the Slash My Bill page. Before a customer can upload a bill and submit it for analysis, the customer must verify ownership of the email address provided. The system sends a 6-digit OTP code to the email, the customer enters the code, and upon successful verification the file upload area and submit button become enabled. This prevents abuse, validates lead quality, and ensures the customer can receive the analysis report.

## Glossary

- **Frontend**: The Slash My Bill single-page application served from `viewMyBill/index.html`, `viewMyBill/viewMyBill.js`, and `viewMyBill/viewMyBill.css`
- **OTP_Lambda**: An AWS Lambda function that handles OTP generation, storage, and verification via API Gateway routes
- **OTP_Table**: A DynamoDB table that stores OTP records with a TTL attribute for automatic expiry
- **SES_Service**: Amazon Simple Email Service used to send OTP emails from a verified sender address on the eshkolai.com domain
- **API_Gateway**: The existing HTTP API v2 (ID: l2fd4h481h) in us-east-1 with a $default stage
- **File_Picker**: The drag-and-drop PDF upload area and its associated file input on the Slash My Bill form
- **Submit_Button**: The "Slash My Bill" button that triggers bill upload and analysis
- **Verify_Button**: A new button labeled "Verify my email" that triggers OTP sending
- **OTP_Input**: A new text input field where the customer enters the 6-digit OTP code
- **OTP_Code**: A 6-digit numeric one-time password generated by OTP_Lambda
- **CloudFormation_Stack**: The existing stack `aws-bill-analyzer-viewmybill` defined in `infrastructure/viewmybill-stack.yaml`

## Requirements

### Requirement 1: Disabled State Before Verification

**User Story:** As a customer, I want the file upload area and submit button to be visually disabled before email verification, so that I understand I must verify my email first.

#### Acceptance Criteria

1. WHEN the Slash My Bill page loads, THE Frontend SHALL render the File_Picker in a disabled state with reduced opacity and no pointer interaction
2. WHEN the Slash My Bill page loads, THE Frontend SHALL render the Submit_Button in a disabled state with reduced opacity and no pointer interaction
3. WHILE the email is not verified, THE Frontend SHALL display a message near the File_Picker indicating that email verification is required before uploading
4. WHILE the email is not verified, THE Frontend SHALL prevent drag-and-drop file operations on the File_Picker

### Requirement 2: Verify Email Button

**User Story:** As a customer, I want a "Verify my email" button that sends an OTP to my email address, so that I can prove I own the email.

#### Acceptance Criteria

1. THE Frontend SHALL display a Verify_Button labeled "Verify my email" below the contact fields section
2. WHILE the Full Name, Email, and Phone fields are empty or invalid, THE Frontend SHALL disable the Verify_Button
3. WHEN the customer clicks the Verify_Button with valid Full Name, Email, and Phone values, THE Frontend SHALL send a POST request to API_Gateway at the `/send-otp` route with the email address in the request body
4. WHEN the customer clicks the Verify_Button, THE Frontend SHALL display a loading indicator on the Verify_Button and disable the Verify_Button until the API response is received
5. WHEN the `/send-otp` request succeeds, THE Frontend SHALL display the OTP_Input field for the customer to enter the code
6. WHEN the `/send-otp` request succeeds, THE Frontend SHALL display a message indicating that an OTP has been sent to the provided email address
7. IF the `/send-otp` request fails, THEN THE Frontend SHALL display an error message describing the failure and re-enable the Verify_Button

### Requirement 3: OTP Generation and Sending

**User Story:** As a system operator, I want OTP codes generated securely and sent via email, so that only the email owner can verify.

#### Acceptance Criteria

1. WHEN the OTP_Lambda receives a `/send-otp` request with a valid email address, THE OTP_Lambda SHALL generate a cryptographically random 6-digit OTP_Code
2. WHEN the OTP_Lambda generates an OTP_Code, THE OTP_Lambda SHALL store the OTP_Code in OTP_Table with the email as the partition key, a creation timestamp, and a TTL attribute set to 5 minutes from creation
3. WHEN the OTP_Lambda stores the OTP_Code, THE OTP_Lambda SHALL send an email via SES_Service from `noreply@eshkolai.com` to the customer email containing the OTP_Code
4. THE OTP_Lambda SHALL include the OTP_Code, a 5-minute validity notice, and the Eshkol AI branding in the email body
5. IF the email address format is invalid, THEN THE OTP_Lambda SHALL return a 400 status code with a descriptive error message
6. IF SES_Service fails to send the email, THEN THE OTP_Lambda SHALL return a 500 status code with a descriptive error message
7. WHEN a new OTP is requested for an email that already has an unexpired OTP, THE OTP_Lambda SHALL overwrite the previous OTP record with the new OTP_Code and reset the TTL

### Requirement 4: OTP Verification

**User Story:** As a customer, I want to enter the OTP code I received to verify my email, so that I can proceed with uploading my bill.

#### Acceptance Criteria

1. WHEN the customer submits the OTP_Code via the OTP_Input, THE Frontend SHALL send a POST request to API_Gateway at the `/verify-otp` route with the email and OTP_Code in the request body
2. WHEN the OTP_Lambda receives a `/verify-otp` request, THE OTP_Lambda SHALL look up the OTP record in OTP_Table by email
3. WHEN the submitted OTP_Code matches the stored OTP_Code and the record has not expired, THE OTP_Lambda SHALL return a 200 status code with a success response
4. IF the submitted OTP_Code does not match the stored OTP_Code, THEN THE OTP_Lambda SHALL return a 400 status code with the message "Invalid OTP code"
5. IF no OTP record exists for the email or the record has expired, THEN THE OTP_Lambda SHALL return a 400 status code with the message "OTP has expired or was not requested"
6. WHEN verification succeeds, THE OTP_Lambda SHALL delete the OTP record from OTP_Table

### Requirement 5: Post-Verification Enabled State

**User Story:** As a customer, I want the file upload and submit button to become active after verification, so that I can upload my bill.

#### Acceptance Criteria

1. WHEN the `/verify-otp` response indicates success, THE Frontend SHALL enable the File_Picker with full opacity and pointer interaction
2. WHEN the `/verify-otp` response indicates success, THE Frontend SHALL remove the verification-required message from the File_Picker area
3. WHEN the `/verify-otp` response indicates success, THE Frontend SHALL enable drag-and-drop file operations on the File_Picker
4. WHEN the `/verify-otp` response indicates success, THE Frontend SHALL hide the OTP_Input and Verify_Button sections
5. WHEN the `/verify-otp` response indicates success, THE Frontend SHALL display a confirmation message indicating the email has been verified
6. WHILE the email is verified and a valid file is selected, THE Frontend SHALL enable the Submit_Button
7. WHEN the customer changes the email field value after successful verification, THE Frontend SHALL reset the verification state to unverified and disable the File_Picker and Submit_Button

### Requirement 6: OTP Rate Limiting

**User Story:** As a system operator, I want to limit OTP requests per email, so that the system is protected from abuse and excessive SES costs.

#### Acceptance Criteria

1. WHEN the OTP_Lambda receives a `/send-otp` request for an email that has received an OTP within the last 60 seconds, THE OTP_Lambda SHALL return a 429 status code with the message "Please wait before requesting a new code"
2. WHEN the Frontend receives a 429 response, THE Frontend SHALL display the rate-limit message and disable the Verify_Button for the remaining cooldown period
3. THE Frontend SHALL display a countdown timer on the Verify_Button showing the seconds remaining before a new OTP can be requested

### Requirement 7: OTP Infrastructure

**User Story:** As a system operator, I want the OTP infrastructure provisioned via CloudFormation, so that it deploys automatically through the existing CI/CD pipeline.

#### Acceptance Criteria

1. THE CloudFormation_Stack SHALL define the OTP_Table as a DynamoDB table with `email` as the partition key, PAY_PER_REQUEST billing, server-side encryption enabled, and a TTL attribute named `ttl`
2. THE CloudFormation_Stack SHALL define the OTP_Lambda as a Python 3.12 Lambda function with a 30-second timeout and 128 MB memory
3. THE CloudFormation_Stack SHALL define an IAM role for OTP_Lambda with permissions to read and write to OTP_Table and to send emails via SES_Service
4. THE CloudFormation_Stack SHALL define two API_Gateway routes: `POST /send-otp` and `POST /verify-otp`, both integrated with OTP_Lambda
5. THE CloudFormation_Stack SHALL define a Lambda invoke permission allowing API_Gateway to invoke OTP_Lambda
6. THE CloudFormation_Stack SHALL include SES domain identity verification for eshkolai.com with Route 53 DNS records for DKIM authentication

### Requirement 8: CI/CD Pipeline Update

**User Story:** As a system operator, I want the OTP Lambda packaged and deployed through the existing GitHub Actions workflow, so that deployments remain automated.

#### Acceptance Criteria

1. WHEN the deploy workflow runs, THE deploy workflow SHALL package the OTP Lambda source code into a zip file and upload the zip to the S3 storage bucket
2. WHEN the deploy workflow runs, THE deploy workflow SHALL update the OTP_Lambda function code from the S3 package after CloudFormation deployment
3. THE deploy workflow SHALL trigger on changes to files in the `otp-handler/` directory in addition to existing trigger paths

### Requirement 9: Resend OTP

**User Story:** As a customer, I want to be able to resend the OTP if I did not receive it, so that I can still verify my email.

#### Acceptance Criteria

1. WHEN the OTP_Input is displayed, THE Frontend SHALL display a "Resend code" link below the OTP_Input
2. WHILE the 60-second cooldown period has not elapsed since the last OTP request, THE Frontend SHALL disable the "Resend code" link and display the remaining seconds
3. WHEN the customer clicks the "Resend code" link after the cooldown period, THE Frontend SHALL send a new `/send-otp` request with the same email address
4. WHEN a resend request succeeds, THE Frontend SHALL display a confirmation message that a new code has been sent

### Requirement 10: CORS Configuration

**User Story:** As a system operator, I want the API Gateway CORS configuration to support the new OTP routes, so that the Frontend can call the OTP endpoints.

#### Acceptance Criteria

1. THE API_Gateway CORS configuration SHALL allow POST and OPTIONS methods for the `/send-otp` and `/verify-otp` routes
2. THE API_Gateway CORS configuration SHALL allow origins `https://eshkolai.com` and `https://www.eshkolai.com`
3. THE OTP_Lambda SHALL include CORS headers (`Access-Control-Allow-Origin`, `Access-Control-Allow-Headers`, `Access-Control-Allow-Methods`) in all responses
