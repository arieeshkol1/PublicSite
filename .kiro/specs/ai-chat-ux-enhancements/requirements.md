# Requirements Document

## Introduction

This feature enhances the AI chat experience in the members portal with two improvements: (1) Data Source Buttons that display collapsible, labeled buttons after each AI response to reveal the tabular data used to generate the answer, and (2) Backend-Generated Follow-up Question Suggestions that replace the current client-side keyword matching with contextual, adaptive questions returned by the AI agent in the API response payload.

## Glossary

- **Chat_UI**: The AI chat interface in the members portal rendered by `addAIMessage()` in `members.js`
- **Data_Source_Button**: A labeled, collapsible button displayed below an AI answer that expands to reveal tabular data used in generating the response
- **Table_Area**: The `.ai-table-area` DOM section within each AI answer message that holds data source buttons and rendered tables
- **Follow_Up_Suggestion**: A contextual follow-up question generated by the backend AI agent based on the actual data analyzed during the query
- **API_Response**: The JSON payload returned by the `/members/accounts/ai-query` endpoint containing the answer and metadata
- **Chart_Data**: The array of structured data objects (charts/tables) built by `_build_chart_data()` from cached account cost data
- **Bedrock_Agent**: The AWS Bedrock Agent that orchestrates FinOps analysis and generates answers
- **Member_Handler**: The backend Lambda function (`member-handler/lambda_function.py`) that processes AI chat requests

## Requirements

### Requirement 1: Data Source Buttons — Collapsed Default State

**User Story:** As a member, I want the data tables behind an AI answer to be collapsed by default, so that the chat remains clean and I can expand only the data I care about.

#### Acceptance Criteria

1. WHEN the Chat_UI renders an AI answer that includes Chart_Data, THE Table_Area SHALL display one Data_Source_Button per chart data item in a collapsed state.
2. THE Data_Source_Button SHALL display a descriptive label matching the title of the corresponding chart data item (e.g., "Daily Costs", "Service Breakdown").
3. THE Data_Source_Button SHALL render in the collapsed state by default, hiding the associated tabular content.
4. WHEN a user clicks a collapsed Data_Source_Button, THE Table_Area SHALL expand the associated table content below the button.
5. WHEN a user clicks an expanded Data_Source_Button, THE Table_Area SHALL collapse the associated table content.

### Requirement 2: Data Source Buttons — Visual Design

**User Story:** As a member, I want data source buttons to be visually distinct and easy to identify, so that I know which data supports each answer.

#### Acceptance Criteria

1. THE Data_Source_Button SHALL include a table icon prefix (📋) and the chart data title as its label text.
2. THE Data_Source_Button SHALL visually indicate its expanded or collapsed state using a directional indicator (▶ for collapsed, ▼ for expanded).
3. THE Data_Source_Button SHALL use the existing `btn btn-outline btn-sm` CSS classes for consistent styling with the current chat interface.
4. WHILE a Data_Source_Button is in the expanded state, THE Data_Source_Button SHALL apply a highlighted border style to distinguish it from collapsed buttons.

### Requirement 3: Data Source Buttons — Table Rendering

**User Story:** As a member, I want to see the actual data tables when I expand a data source button, so that I can verify and explore the numbers behind the AI answer.

#### Acceptance Criteria

1. WHEN a Data_Source_Button is expanded, THE Table_Area SHALL render the corresponding chart data as an HTML table with headers and rows.
2. THE Table_Area SHALL format currency values with a dollar sign prefix and two decimal places.
3. THE Table_Area SHALL render percentage values with one decimal place and a percent sign suffix.
4. IF Chart_Data contains no data items, THEN THE Table_Area SHALL not render any Data_Source_Buttons.
5. WHEN multiple Data_Source_Buttons are present, THE Table_Area SHALL allow independent expand/collapse of each button without affecting other buttons.

### Requirement 4: Follow-up Question Suggestions — Backend Generation

**User Story:** As a member, I want contextual follow-up questions generated by the AI based on the actual data it analyzed, so that I get relevant drill-down options rather than generic keyword-based suggestions.

#### Acceptance Criteria

1. WHEN the Bedrock_Agent completes an analysis, THE Member_Handler SHALL generate 2 to 3 follow-up question suggestions based on the data analyzed during the query.
2. THE Member_Handler SHALL include the follow-up suggestions in the API_Response under a `followUpQuestions` field as an array of strings.
3. THE Member_Handler SHALL generate follow-up questions that reference specific services, cost patterns, or anomalies found in the analyzed data.
4. IF the Bedrock_Agent analysis produces insufficient data context, THEN THE Member_Handler SHALL return an empty `followUpQuestions` array.
5. THE Member_Handler SHALL limit follow-up question text to a maximum of 100 characters per question.

### Requirement 5: Follow-up Question Suggestions — Frontend Rendering

**User Story:** As a member, I want to see AI-generated follow-up suggestions as clickable buttons after each answer, so that I can easily continue my analysis.

#### Acceptance Criteria

1. WHEN the API_Response contains a non-empty `followUpQuestions` array, THE Chat_UI SHALL render each suggestion as a clickable button in the follow-ups area.
2. THE Chat_UI SHALL replace the current client-side keyword-matching follow-up logic with backend-provided suggestions when the `followUpQuestions` field is present in the API_Response.
3. WHEN a user clicks a follow-up suggestion button, THE Chat_UI SHALL submit that question text as a new AI query.
4. THE Chat_UI SHALL fall back to the existing client-side follow-up generation when the `followUpQuestions` field is absent or empty in the API_Response.
5. THE Chat_UI SHALL render follow-up suggestion buttons using the existing `ai-followup-btn` CSS class for consistent styling.

### Requirement 6: Follow-up Question Suggestions — Adaptive Count

**User Story:** As a member, I want the number of follow-up suggestions to adapt based on data availability, so that I only see relevant options without clutter.

#### Acceptance Criteria

1. WHEN rich multi-service data is available in the analysis, THE Member_Handler SHALL generate 3 follow-up questions.
2. WHEN limited single-service data is available in the analysis, THE Member_Handler SHALL generate 2 follow-up questions.
3. THE Member_Handler SHALL not generate more than 3 follow-up questions per response.
4. THE Member_Handler SHALL not generate fewer than 2 follow-up questions when any data context is available.

### Requirement 7: API Response Contract

**User Story:** As a developer, I want a clear API response contract for the new fields, so that frontend and backend remain in sync.

#### Acceptance Criteria

1. THE API_Response SHALL include a `followUpQuestions` field of type array containing zero to three string elements.
2. THE API_Response SHALL continue to include the existing `chartData` field unchanged for backward compatibility.
3. THE API_Response SHALL include a `dataSources` field of type array containing objects with `label` (string) and `data` (array of row objects) for each data table used in the answer.
4. IF the Bedrock_Agent path is used, THEN THE API_Response SHALL include both `followUpQuestions` and `dataSources` fields in the response payload.
5. IF an error occurs during follow-up generation, THEN THE Member_Handler SHALL return the response with an empty `followUpQuestions` array without failing the main answer.
