# Requirements Document

## Introduction

Invoice Drilldown is an enhancement to the existing Invoice Explorer feature within the SlashMyBill member portal. It redesigns the invoice browsing experience from a flat table into a top-down hierarchical drill-down interface with three levels: Invoice → Service → Resource. Members can expand an invoice to see service-level cost breakdowns, then expand a service to see individual resource-level costs with explanations of how each cost is calculated. This feature introduces new data sources including the AWS Billing API (DescribeInvoices) for invoice metadata and Cost Explorer with RESOURCE dimension for resource-level granularity, while building on the existing DynamoDB cache, cross-account role assumption, and member-handler Lambda infrastructure.

## Glossary

- **Invoice_Drilldown_UI**: The hierarchical three-level expandable interface replacing the flat invoice table in the member portal
- **Invoice_Level**: The top level of the hierarchy showing invoice entities with metadata from the AWS Billing API
- **Service_Level**: The middle level showing AWS services within an invoice with cost amounts and calculation explanations
- **Resource_Level**: The bottom level showing individual AWS resources within a service with resource metadata and cost calculations
- **Billing_API**: The AWS Billing API (DescribeInvoices) used to retrieve invoice-level metadata such as invoice ID, payment date, and payment status
- **Cost_Explorer_Resource_API**: AWS Cost Explorer GetCostAndUsage API called with the RESOURCE dimension to retrieve resource-level cost data
- **Resource_Metadata_API**: AWS service-specific Describe APIs (EC2 DescribeInstances, RDS DescribeDBInstances, S3 ListBuckets) used to enrich resource IDs with human-readable names and types
- **Member_Handler_Lambda**: The existing AWS Lambda function that serves member portal API requests
- **Invoice_Cache**: The DynamoDB storage layer for normalized invoice, service, and resource data
- **Cross_Account_Role**: The IAM role (SlashMyBill-{AccountID}) assumed via STS to access customer AWS accounts
- **Cost_Explanation**: A human-readable string describing how a cost is calculated (e.g., "730 hours × $0.0464/hr")
- **AI_Cost_Explanation**: A conversational, plain-language description generated by Amazon Bedrock that explains what was charged and why in terms a non-technical person can understand
- **Payment_Status**: The status of an invoice payment, one of: paid, pending, or overdue
- **Member_Portal_Frontend**: The browser-based SPA served from S3 via CloudFront at the /members/ path

## Requirements

### Requirement 1: Invoice-Level Data Retrieval

**User Story:** As a member, I want to see a list of my AWS invoices with key metadata (invoice ID, issuer, payment date, status, total amount), so that I have a clear overview of my billing history at the highest level.

#### Acceptance Criteria

1. WHEN a member requests the invoice list for a connected account, THE Member_Handler_Lambda SHALL call the Billing_API (DescribeInvoices) via Cross_Account_Role to retrieve invoice-level metadata including invoice ID, issuer, payment date, payment status, and total amount
2. WHEN invoice-level data is retrieved from the Billing_API, THE Member_Handler_Lambda SHALL cache the normalized invoice records in the Invoice_Cache with a TTL of 90 days
3. WHEN cached invoice-level data exists for the requested account and is within the TTL period, THE Member_Handler_Lambda SHALL return the cached data without calling the Billing_API
4. THE Member_Handler_Lambda SHALL return each invoice record containing: invoice ID (string from AWS), issuer (string, default "Amazon Web Services"), payment date (ISO 8601 date), payment status (one of: paid, pending, overdue), and total amount (numeric, 2 decimal places, in USD)
5. IF the Billing_API call fails or returns an error, THEN THE Member_Handler_Lambda SHALL return a descriptive error message indicating the failure reason and SHALL NOT store partial data in the Invoice_Cache
6. IF the Billing_API is not available or not supported for the target account, THEN THE Member_Handler_Lambda SHALL fall back to generating invoice-level records from Cost_Explorer_API monthly aggregations, using the billing month as a synthetic invoice identifier

### Requirement 2: Service-Level Drill-Down

**User Story:** As a member, I want to expand an invoice to see which AWS services contributed to the total cost with explanations of how each service cost is calculated, so that I can understand my spending at the service level.

#### Acceptance Criteria

1. WHEN a member expands an invoice in the Invoice_Drilldown_UI, THE Member_Handler_Lambda SHALL return a list of AWS services within that invoice period, each containing: service name, cost amount (2 decimal places in USD), and a Cost_Explanation string
2. WHEN service-level data for the requested invoice period exists in the Invoice_Cache, THE Member_Handler_Lambda SHALL return the cached data without calling the Cost_Explorer_API
3. WHEN service-level data is not cached, THE Invoice_Sync_Service SHALL fetch it from the Cost_Explorer_API with SERVICE dimension grouping via Cross_Account_Role and store the normalized records in the Invoice_Cache
4. THE Member_Handler_Lambda SHALL generate a Cost_Explanation for each service by combining the usage quantity, unit, and unit rate from the Cost_Explorer_API response (e.g., "730 hours × $0.0464/hr" or "100 GB × $0.023/GB")
5. IF the Cost_Explorer_API does not provide sufficient usage quantity or rate data to construct a meaningful Cost_Explanation, THEN THE Member_Handler_Lambda SHALL return the cost amount with a Cost_Explanation of "See resource breakdown for details"
6. THE Member_Handler_Lambda SHALL sort services within an invoice by cost in descending order by default
7. THE Member_Handler_Lambda SHALL exclude services with a cost below $0.01 from the service-level response to reduce noise

### Requirement 3: Resource-Level Drill-Down

**User Story:** As a member, I want to expand a service to see individual resources (instances, volumes, buckets) with their names, types, and cost calculations, so that I can identify exactly which resources are driving my costs.

#### Acceptance Criteria

1. WHEN a member expands a service in the Invoice_Drilldown_UI, THE Member_Handler_Lambda SHALL return a list of individual resources within that service, each containing: resource name or ID, resource type, Cost_Explanation string, and cost amount (2 decimal places in USD)
2. WHEN resource-level data for the requested service and period exists in the Invoice_Cache, THE Member_Handler_Lambda SHALL return the cached data without calling the Cost_Explorer_Resource_API
3. WHEN resource-level data is not cached, THE Invoice_Sync_Service SHALL fetch it from the Cost_Explorer_Resource_API (GetCostAndUsage with GROUP_BY RESOURCE dimension) via Cross_Account_Role and store the normalized records in the Invoice_Cache
4. THE Member_Handler_Lambda SHALL enrich resource IDs with human-readable names and types by calling the appropriate Resource_Metadata_API (EC2 DescribeInstances, RDS DescribeDBInstances, S3 ListBuckets, or equivalent) via Cross_Account_Role
5. IF the Resource_Metadata_API call fails or the resource no longer exists, THEN THE Member_Handler_Lambda SHALL return the raw resource ID from Cost Explorer as the resource name and "Unknown" as the resource type
6. THE Member_Handler_Lambda SHALL generate a Cost_Explanation for each resource by combining the usage quantity, unit, and unit rate (e.g., "730 hours × $0.0464/hr for t3.medium" or "500 GB-month × $0.08/GB-month")
7. IF the Cost_Explorer_Resource_API does not provide sufficient data to construct a meaningful Cost_Explanation for a resource, THEN THE Member_Handler_Lambda SHALL return the cost amount with a Cost_Explanation of "Flat charge" or the usage type description from Cost Explorer
8. THE Member_Handler_Lambda SHALL sort resources within a service by cost in descending order by default
9. THE Member_Handler_Lambda SHALL apply a TTL of 90 days to all resource-level cache records

### Requirement 4: Hierarchical Navigation UI

**User Story:** As a member, I want an expandable tree-style interface where I can drill down from invoices to services to resources, so that I can navigate my costs intuitively from high-level to granular detail.

#### Acceptance Criteria

1. THE Invoice_Drilldown_UI SHALL render the invoice list as expandable rows where each row can be expanded to reveal the service level, and each service row can be expanded to reveal the resource level
2. WHEN a member clicks an invoice row, THE Invoice_Drilldown_UI SHALL expand that row to show the service-level breakdown below it with a visual indentation indicating the parent-child relationship
3. WHEN a member clicks a service row within an expanded invoice, THE Invoice_Drilldown_UI SHALL expand that row to show the resource-level breakdown below it with further visual indentation
4. WHEN a member clicks an already-expanded row, THE Invoice_Drilldown_UI SHALL collapse that row and hide all nested child rows beneath it
5. THE Invoice_Drilldown_UI SHALL allow multiple invoice rows to be expanded simultaneously so that members can compare services across invoices
6. THE Invoice_Drilldown_UI SHALL allow multiple service rows within the same invoice to be expanded simultaneously
7. WHILE a drill-down level is loading data from the API, THE Invoice_Drilldown_UI SHALL display a loading indicator within the expanded area until the response is received or a 30-second timeout elapses
8. IF loading a drill-down level fails, THEN THE Invoice_Drilldown_UI SHALL display an inline error message with a retry button within the expanded area

### Requirement 5: Invoice-Level Display Columns

**User Story:** As a member, I want to see key invoice metadata at a glance in the top-level view, so that I can quickly identify invoices by date, status, and amount.

#### Acceptance Criteria

1. THE Invoice_Drilldown_UI SHALL display the following columns at the invoice level: Invoice ID, Issued By, Payment Date, Payment Status, and Total Amount
2. THE Invoice_Drilldown_UI SHALL format the Payment Date column as a human-readable date (e.g., "Jan 15, 2025")
3. THE Invoice_Drilldown_UI SHALL display the Payment Status with a color-coded badge: green for "paid", amber for "pending", and red for "overdue"
4. THE Invoice_Drilldown_UI SHALL format the Total Amount column as currency with a dollar sign, thousands separator, and 2 decimal places (e.g., "$1,234.56")
5. THE Invoice_Drilldown_UI SHALL sort invoices by Payment Date in descending order (most recent first) by default
6. WHEN a member clicks a column header at the invoice level, THE Invoice_Drilldown_UI SHALL toggle the sort order for that column between ascending and descending

### Requirement 6: Service-Level Display Columns

**User Story:** As a member, I want to see service names, amounts, and cost explanations when I expand an invoice, so that I understand what each service costs and why.

#### Acceptance Criteria

1. THE Invoice_Drilldown_UI SHALL display the following columns at the service level: Service Name, Amount, and Cost Explanation
2. THE Invoice_Drilldown_UI SHALL format the Amount column as currency with a dollar sign, thousands separator, and 2 decimal places
3. THE Invoice_Drilldown_UI SHALL display the Cost_Explanation as a secondary text line or tooltip that describes how the cost is calculated
4. THE Invoice_Drilldown_UI SHALL display a percentage indicator next to each service amount showing what fraction of the invoice total that service represents (rounded to 1 decimal place)

### Requirement 7: Resource-Level Display Columns

**User Story:** As a member, I want to see resource names, types, cost explanations, and amounts at the deepest level, so that I can pinpoint exactly which resources are costing me money.

#### Acceptance Criteria

1. THE Invoice_Drilldown_UI SHALL display the following columns at the resource level: Resource Name/ID, Resource Type, Cost Explanation, and Amount
2. THE Invoice_Drilldown_UI SHALL format the Amount column as currency with a dollar sign, thousands separator, and 2 decimal places
3. THE Invoice_Drilldown_UI SHALL display the Resource Type in a styled tag or badge (e.g., "t3.medium", "gp3", "Standard")
4. THE Invoice_Drilldown_UI SHALL display the Cost_Explanation as descriptive text showing the calculation formula for that resource
5. IF a resource name is not available and only the resource ID is shown, THEN THE Invoice_Drilldown_UI SHALL display the resource ID in a monospace font to distinguish it from human-readable names

### Requirement 8: Data Caching Strategy for Drill-Down Levels

**User Story:** As a member, I want drill-down data to load quickly after the first access, so that expanding and collapsing levels feels responsive.

#### Acceptance Criteria

1. THE Invoice_Cache SHALL store data at three granularity levels: invoice-level records, service-level records (keyed by invoice period), and resource-level records (keyed by service and period)
2. WHEN a member expands a level that has been previously loaded in the current session, THE Invoice_Drilldown_UI SHALL display the data from the client-side cache without making an additional API call
3. THE Member_Handler_Lambda SHALL apply a TTL of 90 days to all cached records across all three levels
4. WHEN a member triggers a manual refresh, THE Invoice_Sync_Service SHALL invalidate and re-fetch data for all three levels (invoice, service, resource) for the specified account and period
5. IF a refresh request is made for an account that was refreshed within the last 5 minutes, THEN THE Member_Handler_Lambda SHALL return a 429 response including the number of seconds remaining in the cooldown period

### Requirement 9: Cross-Account Data Access for New APIs

**User Story:** As a member, I want the system to securely access my AWS Billing API and resource metadata using the existing cross-account role, so that I get accurate invoice and resource data without additional setup.

#### Acceptance Criteria

1. WHEN accessing the Billing_API or Resource_Metadata_API for a member's account, THE Member_Handler_Lambda SHALL use the existing Cross_Account_Role (SlashMyBill-{AccountID}) via STS AssumeRole with the member's ExternalId
2. IF the Cross_Account_Role does not have permissions for the Billing_API (invoicing:DescribeInvoices), THEN THE Member_Handler_Lambda SHALL return a 403 error advising the member to update their CloudFormation template to include billing read permissions
3. IF the Cross_Account_Role does not have permissions for the Resource_Metadata_API (ec2:DescribeInstances, rds:DescribeDBInstances, etc.), THEN THE Member_Handler_Lambda SHALL return resource data without enrichment (raw resource IDs only) and include a warning message indicating that resource names are unavailable due to missing permissions
4. THE Member_Handler_Lambda SHALL verify account ownership before making any cross-account API call, returning a 403 error if the account does not belong to the authenticated member

### Requirement 10: API Endpoints for Drill-Down Data

**User Story:** As a developer, I want well-defined API endpoints for each drill-down level, so that the frontend can fetch data lazily as the user expands each level.

#### Acceptance Criteria

1. WHEN the invoice-list endpoint is called with a valid accountId, THE Member_Handler_Lambda SHALL return a paginated list of invoice-level records for that account sorted by payment date descending
2. WHEN the service-breakdown endpoint is called with a valid accountId and invoice period identifier, THE Member_Handler_Lambda SHALL return the list of services for that invoice period sorted by cost descending
3. WHEN the resource-breakdown endpoint is called with a valid accountId, invoice period, and service name, THE Member_Handler_Lambda SHALL return the list of resources for that service and period sorted by cost descending
4. THE Member_Handler_Lambda SHALL validate all endpoint parameters (accountId format, period format, service name non-empty) and return a 400 error with a descriptive message for invalid inputs
5. THE Member_Handler_Lambda SHALL support pagination on the invoice-list endpoint with page and pageSize parameters (default pageSize 25, maximum 100)
6. THE Member_Handler_Lambda SHALL require a valid authentication token for all drill-down endpoints and return a 401 error for unauthenticated requests

### Requirement 11: Cost Explanation Generation

**User Story:** As a member, I want to see plain-language explanations of how each cost is calculated, so that I understand my bill without needing to interpret raw AWS billing data.

#### Acceptance Criteria

1. WHEN generating a Cost_Explanation for a service or resource, THE Member_Handler_Lambda SHALL combine the usage quantity, usage unit, and effective unit rate into a formula string in the format "{quantity} {unit} × ${rate}/{unit}" (e.g., "730 hours × $0.0464/hr")
2. WHEN the usage quantity is measured in hours and equals approximately 730 (within ±10), THE Member_Handler_Lambda SHALL annotate the explanation with "(full month)" to indicate continuous usage
3. WHEN a service or resource has multiple usage types contributing to its cost, THE Member_Handler_Lambda SHALL list each usage type as a separate line in the Cost_Explanation with its individual formula and amount
4. IF the Cost_Explorer_API returns a blended or amortized cost without a clear unit rate, THEN THE Member_Handler_Lambda SHALL display the Cost_Explanation as "Monthly charge: ${amount}" without a formula breakdown
5. THE Member_Handler_Lambda SHALL round all monetary values in Cost_Explanations to 4 significant digits for rates (e.g., "$0.0464/hr") and 2 decimal places for totals (e.g., "$33.87")

### Requirement 12: Error Handling for Drill-Down Operations

**User Story:** As a member, I want clear feedback when a drill-down level cannot be loaded, so that I understand what went wrong and can take corrective action.

#### Acceptance Criteria

1. IF the Billing_API returns an access denied error, THEN THE Member_Handler_Lambda SHALL return a 403 error with a message advising the member to update their IAM role permissions to include invoicing:DescribeInvoices
2. IF the Cost_Explorer_Resource_API returns a "resource-level data not available" error, THEN THE Member_Handler_Lambda SHALL return a 400 error indicating that resource-level granularity requires Cost Explorer hourly-level data to be enabled
3. IF a Resource_Metadata_API call times out (exceeds 10 seconds), THEN THE Member_Handler_Lambda SHALL return the resource data with raw IDs and a warning that resource names could not be resolved
4. IF the Cost_Explorer_Resource_API is throttled after retry attempts, THEN THE Member_Handler_Lambda SHALL return a 429 error indicating the member should try again shortly
5. WHEN any drill-down API endpoint returns an error, THE Invoice_Drilldown_UI SHALL display the error message inline within the expanded area without collapsing the parent row
6. IF a member attempts to expand a service that has zero resources in Cost Explorer, THEN THE Member_Handler_Lambda SHALL return an empty list with a 200 status and THE Invoice_Drilldown_UI SHALL display a message indicating no resource-level data is available for this service

### Requirement 13: AI-Powered Cost Explanations

**User Story:** As a member, I want AI-generated plain-language explanations of what I was charged and why, so that I can understand my AWS bill without any cloud expertise.

#### Acceptance Criteria

1. WHEN a member expands a service or resource row, THE Member_Handler_Lambda SHALL send the cost data (service name, usage types, quantities, rates, resource metadata) to Amazon Bedrock (Nova Lite) to generate a human-language explanation of what was charged and why
2. THE AI-generated explanation SHALL describe the charge in conversational language that a non-technical person can understand (e.g., "You ran a medium-sized server (t3.medium) continuously for the entire month. At AWS's rate of $0.0464 per hour, that's 730 hours × $0.0464 = $33.87" instead of just "730 hours × $0.0464/hr")
3. WHEN the AI explanation is generated, THE Member_Handler_Lambda SHALL cache the explanation in the Invoice_Cache alongside the cost data with the same TTL (90 days) to avoid repeated AI calls for the same data
4. IF the AI explanation has already been cached for a given service or resource record, THEN THE Member_Handler_Lambda SHALL return the cached explanation without calling Bedrock
5. THE AI prompt SHALL include context about the service type, usage patterns, and pricing model so that the explanation can mention relevant details (e.g., "This is a pay-per-use charge for storing 50GB of files in S3 Standard storage class")
6. IF the Bedrock API call fails or times out (exceeds 15 seconds), THEN THE Member_Handler_Lambda SHALL fall back to the formula-based Cost_Explanation from Requirement 11 and display it without the AI enhancement
7. THE Invoice_Drilldown_UI SHALL display the AI explanation in a distinct visual style (e.g., a light-blue info box with an AI icon) below the formula-based explanation to differentiate human-language context from raw calculation data
8. THE Member_Handler_Lambda SHALL limit AI explanation generation to services or resources with a cost above $1.00 to avoid unnecessary Bedrock API calls for trivial charges
9. WHEN generating AI explanations for multiple resources within the same service expansion, THE Member_Handler_Lambda SHALL batch the resource data into a single Bedrock call and parse individual explanations from the response to minimize API calls and latency
