# Requirements Document

## Introduction

The FinOps Settings Healthcheck feature adds a new "FinOps Settings" section to the existing Configure tab in the SlashMyBill Member Portal. This section audits each connected AWS account's billing and cost management settings against FinOps best practices and presents a scored checklist. The system detects whether each account is a management (payer) account or a linked account by calling `organizations:DescribeOrganization` and comparing the `MasterAccountId` with the connected account ID, then displays a different checklist tailored to each account type. Users can fix misconfigured settings directly from the portal via API calls through the existing cross-account role (`SlashMyBill-{accountId}`), or receive guidance for settings that require manual action. The cross-account role template is updated to include the write permissions needed for automated fixes.

## Glossary

- **Healthcheck_API**: The backend Lambda endpoints in `member-handler/lambda_function.py` that scan and remediate FinOps settings for a connected AWS account via the cross-account role
- **Healthcheck_UI**: The frontend section within the Configure tab that displays the FinOps settings checklist, score, and fix controls
- **Management_Account**: An AWS account identified as the organization's payer account, where `MasterAccountId` from `organizations:DescribeOrganization` matches the connected account ID
- **Linked_Account**: An AWS account that is a member of an AWS Organization but is not the management (payer) account
- **Checklist_Item**: A single FinOps best-practice setting with a name, current status (pass/warning/fail), description, and an optional fix action
- **Settings_Score**: A summary metric showing the count of passing checklist items out of the total applicable items for the account type (e.g., "7/9 configured")
- **CrossAccount_Role**: The existing IAM role `SlashMyBill-{accountId}` in the member's AWS account used for cross-account API calls
- **CF_Template**: The CloudFormation YAML template generated by `handle_generate_template` that creates the CrossAccount_Role
- **Member_Token**: The JWT token used to authenticate Member Portal API requests
- **Configure_Tab**: The existing tab in the Member Portal (data-tab="accounts-tab") that displays the accounts table and connection management
- **Cost_Allocation_Tag**: An AWS tag key activated for cost allocation reporting in the AWS Billing console
- **Anomaly_Monitor**: An AWS Cost Anomaly Detection monitor that tracks cost anomalies by service dimension
- **Anomaly_Subscription**: An alert subscription attached to an Anomaly_Monitor that sends notifications via email
- **CUR_Report**: An AWS Cost and Usage Report definition that exports detailed billing data to S3
- **Tag_Backfill**: The process of retroactively applying cost allocation tag data to historical billing records (up to 12 months)

## Requirements

### Requirement 1: Account Type Detection

**User Story:** As a member, I want the system to automatically detect whether my connected AWS account is a management account or a linked account, so that I see the correct checklist for my account type.

#### Acceptance Criteria

1. WHEN the member selects an account for healthcheck scanning, THE Healthcheck_API SHALL call `organizations:DescribeOrganization` using the CrossAccount_Role for that account
2. WHEN the `MasterAccountId` returned by `organizations:DescribeOrganization` matches the connected account ID, THE Healthcheck_API SHALL classify the account as a Management_Account
3. WHEN the `MasterAccountId` returned by `organizations:DescribeOrganization` does not match the connected account ID, THE Healthcheck_API SHALL classify the account as a Linked_Account
4. IF the `organizations:DescribeOrganization` call fails with AccessDeniedException, THEN THE Healthcheck_API SHALL classify the account as a Linked_Account and include a note that organization data is unavailable
5. THE Healthcheck_UI SHALL display a badge indicating the detected account type: "👑 Management Account" for Management_Account or "🔗 Linked Account" for Linked_Account

### Requirement 2: Management Account — Cost Allocation Tags (User-Defined)

**User Story:** As a member with a management account, I want to see which user-defined cost allocation tags are active and activate inactive ones, so that my cost reports are properly categorized.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `ce:ListCostAllocationTags` with `Type=UserDefined` using the CrossAccount_Role to retrieve all user-defined cost allocation tags
2. THE Healthcheck_UI SHALL display each discovered user-defined tag with its current status (Active or Inactive)
3. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when all discovered user-defined tags are active, as warning when some tags are inactive, and as fail when no user-defined tags exist
4. WHEN the member clicks "Activate" on inactive tags, THE Healthcheck_API SHALL call `ce:UpdateCostAllocationTagsStatus` using the CrossAccount_Role to activate the selected tags
5. WHEN tag activation succeeds, THE Healthcheck_UI SHALL update the tag status to Active and recalculate the Settings_Score without requiring a full rescan
6. THE Healthcheck_UI SHALL display best-practice guidance recommending activation of Environment, Owner, CostCenter, and Application tags

### Requirement 3: Management Account — AWS-Generated Tags

**User Story:** As a member with a management account, I want to verify that the `aws:createdBy` tag is active, so that I can track resource accountability in cost reports.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `ce:ListCostAllocationTags` with `Type=AWSGenerated` using the CrossAccount_Role to check the status of AWS-generated tags
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when the `aws:createdBy` tag is active, and as fail when the `aws:createdBy` tag is inactive or absent
3. WHEN the member clicks "Activate" on the `aws:createdBy` tag, THE Healthcheck_API SHALL call `ce:UpdateCostAllocationTagsStatus` using the CrossAccount_Role to activate the tag
4. WHEN activation succeeds, THE Healthcheck_UI SHALL update the status to Active and recalculate the Settings_Score

### Requirement 4: Management Account — Cost Anomaly Detection

**User Story:** As a member with a management account, I want to check if cost anomaly detection is configured and set it up if missing, so that I receive alerts on unexpected spending.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `ce:GetAnomalyMonitors` using the CrossAccount_Role to check for existing anomaly monitors
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one Anomaly_Monitor exists, and as fail when no monitors exist
3. WHEN the member clicks "Setup" on this Checklist_Item, THE Healthcheck_API SHALL call `ce:CreateAnomalyMonitor` with type DIMENSIONAL and dimensionValue SERVICE using the CrossAccount_Role
4. WHEN the Anomaly_Monitor is created, THE Healthcheck_API SHALL call `ce:CreateAnomalySubscription` to create an email alert subscription linked to the new monitor
5. WHEN monitor and subscription creation succeed, THE Healthcheck_UI SHALL update the status to pass and recalculate the Settings_Score
6. IF the `ce:CreateAnomalyMonitor` or `ce:CreateAnomalySubscription` call fails, THEN THE Healthcheck_API SHALL return a descriptive error message and THE Healthcheck_UI SHALL display the error to the member

### Requirement 5: Management Account — Hourly Granularity

**User Story:** As a member with a management account, I want to know if hourly cost granularity is enabled, so that I can get real-time cost visibility.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL probe `ce:GetCostAndUsage` with HOURLY granularity using the CrossAccount_Role to determine if hourly data is available
2. WHEN the HOURLY probe returns data, THE Healthcheck_UI SHALL mark this Checklist_Item as pass
3. WHEN the HOURLY probe returns an error or empty result indicating hourly granularity is not enabled, THE Healthcheck_UI SHALL mark this Checklist_Item as fail
4. THE Healthcheck_UI SHALL display guidance text explaining that hourly granularity cannot be enabled via API and must be enabled manually in the AWS Billing console

### Requirement 6: Management Account — Cost Explorer Preferences (Right-Sizing)

**User Story:** As a member with a management account, I want to check if EC2 rightsizing recommendations are enabled and enable them if not, so that I receive optimization suggestions.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `ce:GetPreferences` using the CrossAccount_Role to check if RightsizingRecommendations is enabled
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when RightsizingRecommendations is enabled, and as fail when RightsizingRecommendations is disabled
3. WHEN the member clicks "Enable" on this Checklist_Item, THE Healthcheck_API SHALL call `ce:UpdatePreferences` using the CrossAccount_Role to enable RightsizingRecommendations
4. WHEN the preference update succeeds, THE Healthcheck_UI SHALL update the status to pass and recalculate the Settings_Score

### Requirement 7: Management Account — Cost and Usage Report (CUR)

**User Story:** As a member with a management account, I want to verify that a Cost and Usage Report is configured, so that I have detailed billing data for analysis.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `cur:DescribeReportDefinitions` using the CrossAccount_Role to check for existing CUR reports
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one CUR_Report definition exists, and as fail when no CUR reports are defined
3. THE Healthcheck_UI SHALL display guidance text explaining that creating a CUR report requires S3 bucket setup and may need to be configured in the AWS console
4. THE Healthcheck_UI SHALL display the names and S3 destinations of existing CUR reports when they exist

### Requirement 8: Management Account — Tag Backfill

**User Story:** As a member with a management account, I want to check if tag backfill has been run and trigger it if needed, so that historical cost data reflects newly activated tags.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `ce:ListCostAllocationTagBackfillHistory` using the CrossAccount_Role to check if a backfill has been executed
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one completed backfill exists, as warning when a backfill is in progress, and as fail when no backfill history exists
3. WHEN the member clicks "Start Backfill" on this Checklist_Item, THE Healthcheck_API SHALL call `ce:StartCostAllocationTagBackfill` using the CrossAccount_Role to initiate a backfill of up to 12 months
4. WHEN the backfill request succeeds, THE Healthcheck_UI SHALL update the status to warning with a message indicating backfill is in progress
5. IF the `ce:StartCostAllocationTagBackfill` call fails, THEN THE Healthcheck_API SHALL return a descriptive error message and THE Healthcheck_UI SHALL display the error to the member

### Requirement 9: Management Account — Linked Account Billing Access

**User Story:** As a member with a management account, I want to check if linked accounts have access to billing data, so that team members in linked accounts can view their costs.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL check the organization settings to determine if linked account billing access is enabled
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when linked account billing access is enabled, and as fail when it is disabled
3. THE Healthcheck_UI SHALL display guidance text explaining that linked account billing access must be enabled in the AWS Organizations console and cannot be changed via API

### Requirement 10: Management Account — Budgets

**User Story:** As a member with a management account, I want to check if AWS Budgets are configured, so that I have spending alerts in place.

#### Acceptance Criteria

1. WHEN scanning a Management_Account, THE Healthcheck_API SHALL call `budgets:DescribeBudgets` using the CrossAccount_Role to check for existing budgets
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one budget exists, and as fail when no budgets are defined
3. THE Healthcheck_UI SHALL display a link to the Plan → Budget section in the Member Portal for creating budgets
4. THE Healthcheck_UI SHALL display best-practice guidance recommending at least one monthly budget with alerts at 50%, 75%, and 100% thresholds

### Requirement 11: Linked Account — Resource Tag Coverage

**User Story:** As a member with a linked account, I want to see what percentage of my resources are tagged, so that I can improve cost attribution.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL call `tag:GetResources` using the CrossAccount_Role to count tagged and untagged resources
2. THE Healthcheck_UI SHALL display the tag coverage percentage (tagged resources divided by total resources)
3. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when tag coverage exceeds 80%, as warning when coverage is between 50% and 80%, and as fail when coverage is below 50%
4. THE Healthcheck_UI SHALL display a link to the Plan → Tag Resources section in the Member Portal for tagging resources

### Requirement 12: Linked Account — Budgets

**User Story:** As a member with a linked account, I want to check if budgets are configured for my account, so that I have spending alerts.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL call `budgets:DescribeBudgets` using the CrossAccount_Role to check for existing budgets
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one budget exists, and as fail when no budgets are defined
3. THE Healthcheck_UI SHALL display a link to the Plan → Budget section in the Member Portal for creating budgets

### Requirement 13: Linked Account — Cost Anomaly Detection

**User Story:** As a member with a linked account, I want to check if cost anomaly detection is configured and set it up if missing, so that I receive alerts on unexpected spending.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL call `ce:GetAnomalyMonitors` using the CrossAccount_Role to check for existing anomaly monitors
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when at least one Anomaly_Monitor exists, and as fail when no monitors exist
3. WHEN the member clicks "Setup" on this Checklist_Item, THE Healthcheck_API SHALL call `ce:CreateAnomalyMonitor` with type DIMENSIONAL and dimensionValue SERVICE, followed by `ce:CreateAnomalySubscription` for email alerts, using the CrossAccount_Role
4. WHEN creation succeeds, THE Healthcheck_UI SHALL update the status to pass and recalculate the Settings_Score

### Requirement 14: Linked Account — Compute Optimizer Enrollment

**User Story:** As a member with a linked account, I want to check if AWS Compute Optimizer is enrolled and enroll if not, so that I receive rightsizing recommendations.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL call `compute-optimizer:GetEnrollmentStatus` using the CrossAccount_Role to check enrollment status
2. THE Healthcheck_UI SHALL mark this Checklist_Item as pass when the enrollment status is Active, and as fail when the status is Inactive or NotEnrolled
3. WHEN the member clicks "Enroll" on this Checklist_Item, THE Healthcheck_API SHALL call `compute-optimizer:UpdateEnrollmentStatus` with status Active using the CrossAccount_Role
4. WHEN enrollment succeeds, THE Healthcheck_UI SHALL update the status to pass and recalculate the Settings_Score

### Requirement 15: Linked Account — Hourly Granularity

**User Story:** As a member with a linked account, I want to know if hourly cost granularity is available for my account, so that I understand my real-time cost visibility.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL probe `ce:GetCostAndUsage` with HOURLY granularity using the CrossAccount_Role
2. WHEN the HOURLY probe returns data, THE Healthcheck_UI SHALL mark this Checklist_Item as pass
3. WHEN the HOURLY probe fails or returns empty results, THE Healthcheck_UI SHALL mark this Checklist_Item as fail
4. THE Healthcheck_UI SHALL display guidance text stating "Must be enabled from the management account" when hourly granularity is not available

### Requirement 16: Linked Account — Tag Activation Status (Read-Only)

**User Story:** As a member with a linked account, I want to see the status of cost allocation tags, so that I know if my management account admin has activated them.

#### Acceptance Criteria

1. WHEN scanning a Linked_Account, THE Healthcheck_API SHALL attempt to call `ce:ListCostAllocationTags` using the CrossAccount_Role
2. WHEN the call succeeds, THE Healthcheck_UI SHALL display the list of cost allocation tags and their activation status
3. WHEN the call fails with AccessDeniedException, THE Healthcheck_UI SHALL display the message "Managed by payer account" and mark this Checklist_Item as warning
4. THE Healthcheck_UI SHALL display guidance text stating "Ask your management account admin to activate cost allocation tags" when tags are not active or access is denied

### Requirement 17: FinOps Settings UI Layout

**User Story:** As a member, I want the FinOps Settings section to be clearly organized within the Configure tab, so that I can quickly assess and improve my account's FinOps posture.

#### Acceptance Criteria

1. THE Healthcheck_UI SHALL appear as a new sub-section within the Configure_Tab, accessible via a left-nav item labeled "FinOps Settings" alongside the existing accounts management content
2. THE Healthcheck_UI SHALL display an account selector dropdown at the top, populated with all connected accounts that have a "connected" connection status
3. THE Healthcheck_UI SHALL display the Settings_Score as "X/Y FinOps settings configured" where X is the count of passing items and Y is the total checklist items for the detected account type
4. THE Healthcheck_UI SHALL display each Checklist_Item with: a status icon (✅ for pass, ⚠️ for warning, ❌ for fail), the setting name, the current status description, best-practice guidance, and a Fix/Setup/Enable button where an automated fix is available
5. WHEN a Checklist_Item cannot be fixed via API, THE Healthcheck_UI SHALL display guidance text with instructions or a link instead of a fix button
6. THE Healthcheck_UI SHALL display a "Scan Settings" button that triggers a full rescan of all checklist items for the selected account
7. WHEN a scan or fix operation is in progress, THE Healthcheck_UI SHALL display a loading indicator on the affected Checklist_Item or on the entire section during a full scan

### Requirement 18: Healthcheck Scan API

**User Story:** As a developer, I want a single API endpoint that scans all FinOps settings for an account, so that the frontend can retrieve the full checklist in one call.

#### Acceptance Criteria

1. THE Healthcheck_API SHALL expose a new route `POST /members/healthcheck/scan` that accepts an accountId in the request body and requires a valid Member_Token
2. WHEN the scan endpoint is called, THE Healthcheck_API SHALL assume the CrossAccount_Role for the specified account, detect the account type, and execute all applicable checklist checks in sequence
3. THE Healthcheck_API SHALL return a JSON response containing: the detected account type (management or linked), the list of Checklist_Items with name, status (pass/warning/fail), description, and available fix action, and the Settings_Score
4. IF the CrossAccount_Role assumption fails, THEN THE Healthcheck_API SHALL return a 403 status with a message indicating the connection must be re-established
5. IF an individual checklist check fails due to an API error, THEN THE Healthcheck_API SHALL mark that Checklist_Item as "error" with the error message and continue scanning the remaining items
6. FOR ALL connected accounts with a valid CrossAccount_Role, scanning then re-scanning the same account SHALL produce equivalent Checklist_Item statuses when no settings have changed between scans (idempotence property)

### Requirement 19: Healthcheck Fix API

**User Story:** As a developer, I want individual fix endpoints for each remediable setting, so that the frontend can trigger fixes independently.

#### Acceptance Criteria

1. THE Healthcheck_API SHALL expose a new route `POST /members/healthcheck/fix` that accepts an accountId and a fixAction identifier in the request body and requires a valid Member_Token
2. THE Healthcheck_API SHALL support the following fixAction values: `activate_user_tags`, `activate_aws_tags`, `create_anomaly_monitor`, `enable_rightsizing`, `start_tag_backfill`, `enroll_compute_optimizer`
3. WHEN a fix endpoint is called, THE Healthcheck_API SHALL assume the CrossAccount_Role for the specified account and execute the corresponding AWS API call
4. WHEN the fix operation succeeds, THE Healthcheck_API SHALL return a 200 status with the updated Checklist_Item status
5. IF the fix operation fails, THEN THE Healthcheck_API SHALL return an appropriate error status (400 for invalid request, 403 for permission denied, 500 for AWS API error) with a descriptive error message
6. THE Healthcheck_API SHALL verify that the requested fixAction is applicable to the detected account type before executing the fix

### Requirement 20: Cross-Account Role Template Update

**User Story:** As a developer, I want the CloudFormation template to include write permissions for FinOps settings remediation, so that the healthcheck fix actions work through the cross-account role.

#### Acceptance Criteria

1. THE CF_Template generated by `handle_generate_template` SHALL include the following additional IAM permissions in the SlashMyBillBillingAccess inline policy: `ce:UpdateCostAllocationTagsStatus`, `ce:CreateAnomalyMonitor`, `ce:CreateAnomalySubscription`, `ce:StartCostAllocationTagBackfill`, `ce:GetAnomalyMonitors`, `ce:GetAnomalySubscriptions`, `ce:ListCostAllocationTagBackfillHistory`, `compute-optimizer:GetEnrollmentStatus`, `compute-optimizer:UpdateEnrollmentStatus`, `organizations:DescribeOrganization`
2. THE CF_Template SHALL maintain all existing permissions in the SlashMyBillBillingAccess policy without removing or modifying them
3. WHEN a member generates a new CF_Template after this update, THE CF_Template SHALL include both the existing permissions and the new healthcheck permissions
4. THE Healthcheck_UI SHALL display a notification to members whose accounts were connected before this update, advising them to update their CloudFormation stack to enable fix actions

### Requirement 21: Knowledge Base Tips Integration

**User Story:** As a member, I want the FinOps healthcheck items reflected in the tips knowledge base, so that the AI chat and scan engine can reference them in recommendations.

#### Acceptance Criteria

1. WHEN the feature is finalized, THE knowledge base (`knowledge-base/aws-cost-optimization-tips.json`) SHALL include a new tip for each healthcheck item with category "finops-settings", service "General", and `implementedInHealthcheck: true`
2. Each healthcheck tip SHALL include an `automatedCheck` field describing the API call used to verify the setting (e.g., `ce:ListCostAllocationTags → check active vs inactive count`)
3. Each healthcheck tip SHALL include an `actionType` of "deep-link" with `actionTarget` set to "configure:finops-settings" so that clicking the tip navigates to the FinOps Settings section
4. Each healthcheck tip SHALL include an `accountTypeScope` field set to "management", "linked", or "both" indicating which account types the tip applies to
5. THE seed script (`knowledge-base/seed-dynamodb.py`) SHALL include the new healthcheck tips when seeding the DynamoDB table

### Requirement 22: AI Chat Awareness of FinOps Settings

**User Story:** As a member, I want the AI chat to know about my account's FinOps settings status, so that it can recommend enabling missing settings when relevant.

#### Acceptance Criteria

1. WHEN the AI gathers account data for a query, THE data gathering pipeline SHALL include the latest healthcheck scan results (if available) from the member's DynamoDB record
2. WHEN the AI detects that cost allocation tags are not activated and the member asks about cost-by-tag or cost allocation, THE AI SHALL recommend "Go to Configure → FinOps Settings to activate cost allocation tags"
3. WHEN the AI detects that no anomaly monitors exist and the member asks about cost anomalies or unexpected spending, THE AI SHALL recommend "Go to Configure → FinOps Settings to set up Cost Anomaly Detection"
4. WHEN the AI detects that Compute Optimizer is not enrolled and the member asks about rightsizing, THE AI SHALL recommend "Go to Configure → FinOps Settings to enroll in Compute Optimizer"
5. THE Bedrock prompt's SLASHMYBILL PLATFORM FEATURES section SHALL include "Configure → FinOps Settings: Check and fix AWS billing best practices (cost allocation tags, anomaly detection, rightsizing, hourly granularity)"
6. THE AI SHALL NEVER recommend the member to open the AWS Billing Console for settings that can be fixed via Configure → FinOps Settings

### Requirement 23: Observe Dashboard — FinOps Score Widget

**User Story:** As a member, I want to see my FinOps settings score on the Observe dashboard, so that I have visibility into my account's configuration health at a glance.

#### Acceptance Criteria

1. THE Observe dashboard SHALL include a "FinOps Score" KPI card showing the Settings_Score (e.g., "7/10") for the primary connected account
2. THE FinOps Score KPI card SHALL use color coding: green (≥80%), amber (50-79%), red (<50%)
3. WHEN the member clicks the FinOps Score KPI card, THE portal SHALL navigate to Configure → FinOps Settings
4. IF no healthcheck scan has been run yet for the account, THE FinOps Score KPI card SHALL display "Not scanned" with a "Scan →" link

### Requirement 24: Act Tab — Scan Integration

**User Story:** As a member, I want the waste scan to surface FinOps settings issues alongside resource-level findings, so that I get a complete picture of optimization opportunities.

#### Acceptance Criteria

1. WHEN the Act tab waste scan runs, THE scan engine SHALL check if a recent healthcheck scan exists (within the last 24 hours) for the scanned accounts
2. IF the healthcheck scan shows failing items, THE scan results SHALL include a summary card titled "⚙️ FinOps Settings" showing the count of failing settings and the Settings_Score
3. THE FinOps Settings summary card SHALL display a "Fix in Settings →" button that navigates to Configure → FinOps Settings
4. THE FinOps Settings summary card SHALL list the top 3 failing items by priority (cost allocation tags, anomaly detection, and Compute Optimizer enrollment)
5. IF no healthcheck scan exists, THE FinOps Settings card SHALL display "Run a FinOps Settings scan in Configure → FinOps Settings"

### Requirement 25: Healthcheck Results Persistence

**User Story:** As a member, I want my healthcheck scan results stored, so that the dashboard, AI chat, and Act tab can reference them without re-scanning.

#### Acceptance Criteria

1. WHEN a healthcheck scan completes, THE Healthcheck_API SHALL store the scan results in the member's DynamoDB record under a `healthcheckResults` attribute, keyed by accountId
2. Each stored result SHALL include: accountId, accountType (management/linked), scanTimestamp, settingsScore, and the full list of Checklist_Items with their statuses
3. WHEN a fix action succeeds, THE Healthcheck_API SHALL update the stored result for the affected Checklist_Item without requiring a full rescan
4. THE stored results SHALL be used by the AI chat data gathering pipeline (Requirement 22), the Observe dashboard KPI card (Requirement 23), and the Act tab scan integration (Requirement 24)
