2. Complete User Flow — Step by Step

Step 1: User Visits Page

URL: https://www.eshkolai.com/slashMyBill/

CloudFront serves static files from S3 bucket www.eshkolai.com.

Step 2: Email Verification (OTP)

Step 2.1: User enters email, clicks "Send Code"
Step 2.2: Frontend calls POST /send-otp
Step 2.3: OTP Lambda generates 6-digit code, stores in DynamoDB (TTL: 10 min)
Step 2.4: OTP Lambda sends code via SES from SlashMyBill <noreply@eshkolai.com> with SlashMyBill logo in email
Step 2.5: User enters code, clicks "Verify"
Step 2.6: Frontend calls POST /verify-otp, Lambda checks DynamoDB
Step 2.7: On success, file upload area becomes enabled

Step 3: Bill Upload

Step 3.1: User selects PDF file (max 10MB), clicks "Slash My Bill"
Step 3.2: Frontend sends file as base64 to POST /upload
Step 3.3: Upload Lambda validates file (PDF only, max 10MB, not empty)
Step 3.4: Generates unique session ID (UUID)
Step 3.5: Stores PDF in S3 at bills/{sessionId}/{filename}
Step 3.6: Saves lead to DynamoDB: email, name, company, phone, fileName, timestamp
Step 3.7: Returns {sessionId} to frontend

Step 4: Bill Analysis (Async with Polling)

Frontend calls POST /analyze with {sessionId, email}

Step 4.1: Trigger Async Processing (Sync Lambda, <1 second)

Step 4.1.1: Lambda checks S3 for reports/{sessionId}/result.json — if exists, return result immediately
Step 4.1.2: Lambda checks S3 for reports/{sessionId}/processing marker — if exists, return 202 {status: "processing"}
Step 4.1.3: Creates "processing" marker in S3: reports/{sessionId}/processing
Step 4.1.4: Invokes itself asynchronously: lambda_client.invoke(InvocationType='Event')
Step 4.1.5: Returns 202 {status: "processing", sessionId} to API Gateway → frontend

CloudWatch log trace:

STEP 4.1: Processing marker set, invoking self async for session {id} STEP 4.1 DONE: Self-invoke returned StatusCode=202 for session {id}

Step 4.2: Background Processing (Async Lambda, up to 15 minutes)

This runs in a separate Lambda invocation triggered by the self-invoke. It has its own CloudWatch log stream.

Step 4.2.1: Retrieve PDF from S3
Calls s3.list_objects_v2 + s3.get_object for bills/{sessionId}/
File: lambda_function.py → _retrieve_bill_from_s3()
Step 4.2.2: Parse bill with pdfplumber
Opens PDF, extracts text/tables, parses line items, aggregates service totals, extracts metadata (invoice#, account, period, currency), extracts region/instance details
File: bill_parser.py → parse_bill()
Step 4.2.3: Query DynamoDB for cost optimization tips
Maps bill service names to DynamoDB keys, queries tips for each service + "General", caps at 10 tips
File: bedrock_client.py → _get_optimization_tips()
Step 4.2.4: Call Amazon Bedrock (Nova Lite) for AI analysis
Builds prompt with bill data + tips + eligibility rules. For >20 services, splits into batches. Each batch = 1 Bedrock call. Retries 2x on throttling/503.
File: bedrock_client.py → analyze_bill() → _invoke_bedrock_with_retry()
Step 4.2.5: Generate PDF report
Creates analysis pages with ReportLab (logo, summary, pie chart, per-service sections, recommendations). Merges with original invoice via PyPDF2.
File: pdf_generator.py → generate_report()
Step 4.2.6: Upload report to S3
Uploads PDF to reports/{sessionId}/report.pdf, generates pre-signed URL (24h expiry)
File: lambda_function.py → _upload_report_to_s3()
Step 4.2.7: Save result metadata to S3
Saves reports/{sessionId}/result.json with {downloadUrl, summary, originalFilename}
File: lambda_function.py → _save_result()
Step 4.2.8: Clear processing marker
Deletes reports/{sessionId}/processing from S3
File: lambda_function.py → _clear_processing()

Step 4.3: Frontend Polling

Step 4.3.1: Frontend polls POST /analyze every 5 seconds with same sessionId
Step 4.3.2: Lambda checks for result.json → not found → checks processing marker → found → returns 202
Step 4.3.3: When result.json appears → returns 200 {status: "complete", downloadUrl, summary}
Step 4.3.4: Frontend shows summary text and "Download Report" button
Step 4.3.5: If polling exceeds 10 minutes → frontend shows timeout error
Why async? API Gateway HTTP API has a hard 29-second timeout and kills the Lambda when it times out. The self-invoke pattern (InvocationType='Event') creates a completely separate Lambda execution that runs independently for up to 15 minutes.

Step 5: Report Download

Step 5.1: User clicks "Download Report" button
Step 5.2: Opens pre-signed S3 URL in a new browser tab
Step 5.3: PDF contains: executive summary with savings estimates (monthly/yearly), pie chart, per-service analysis with savings breakdown, recommendations, Savings Plans analysis, original invoice pages
Step 5.4: Pre-signed URL expires after 24 hours
Step 5.5: S3 lifecycle rule deletes bill and report files after 24 hours

3. AI Analysis Pipeline Detail

3.1 Bill Parsing (Step 4.2.2)

File: bill_parser.py

3.1.1: Open PDF with pdfplumber
3.1.2: Detect and skip report pages (re-uploaded analyzed bills)
3.1.3: Extract text and tables from each page
3.1.4: Parse line items: service, description, quantity, rate, amount
3.1.5: Deduplicate line items
3.1.6: Aggregate service totals
3.1.7: Extract metadata: invoice#, account ID, billing period, currency
3.1.8: Extract region and instance type details per service

3.2 Bedrock AI Analysis (Steps 4.2.3-4)

File: bedrock_client.py

3.2.1: Map bill service names to DynamoDB keys (e.g., "Amazon Elastic Compute Cloud" → "EC2")
3.2.2: Query DynamoDB for tips per service + "General" (cap: 10 tips)
3.2.3: Build prompt with: bill data, tips, SP/RI eligibility rules, rightsizing rules
3.2.4: If ≤20 services → single Bedrock call. If >20 → split into batches
3.2.5: Call Bedrock Nova Lite (amazon.nova-lite-v1:0), MAX_TOKENS=4000
3.2.6: Retry logic: 2 retries, exponential backoff (1s, 2s) on 429/503
3.2.7: Parse JSON response: summary, service_analysis[], savings_plan_analysis
3.2.8: If batched → merge results from all batches

3.3 PDF Report Generation (Step 4.2.5)

File: pdf_generator.py

3.3.1: Load SlashMyBill.png logo
3.3.2: Create title page with logo, date, account info
3.3.3: Render bill summary section with total cost
3.3.4: Compute and render savings estimates (monthly and yearly min/max/avg) from AI recommendations
3.3.5: Render pie chart of service costs
3.3.6: Render per-service analysis sections (explanation, billing details, recommendations, per-service monthly/yearly savings)
3.3.7: Render Savings Plans analysis section
3.3.8: Merge analysis pages with original invoice pages (PyPDF2)

Note: All cost values are formatted with comma separators for thousands (e.g., $1,234.56).
